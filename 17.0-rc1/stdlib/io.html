<h1>io</h1> <h2 class="modsummary">Standard I/O Server Interface Functions</h2> <div class="description"> <p>This module provides an interface to standard Erlang I/O servers. The output functions all return <code>ok</code> if they are successful, or exit if they are not.</p> <p>In the following description, all functions have an optional parameter <code>IoDevice</code>. If included, it must be the pid of a process which handles the IO protocols. Normally, it is the <code>IoDevice</code> returned by <a href="../kernel/file.html#open/2" class="seealso">file:open/2</a>.</p> <p>For a description of the IO protocols refer to the <a href="io_protocol.html" class="seealso">STDLIB User's Guide</a>.</p> <div class="warning"><h2>Warning!</h2> <p>As of R13A, data supplied to the <a href="#put_chars/2" class="seealso">put_chars</a> function should be in the <a href="unicode.html#type-chardata" class="seealso"><code>unicode:chardata()</code></a> format. This means that programs supplying binaries to this function need to convert them to UTF-8 before trying to output the data on an IO device.</p> <p>If an IO device is set in binary mode, the functions <a href="#get_chars/3" class="seealso">get_chars</a> and <a href="#get_line/2" class="seealso">get_line</a> may return binaries instead of lists. The binaries will, as of R13A, be encoded in UTF-8.</p> <p>To work with binaries in ISO-latin-1 encoding, use the <a href="../kernel/file.html" class="seealso">file</a> module instead.</p> <p>For conversion functions between character encodings, see the <a href="../stdlib/unicode.html" class="seealso">unicode</a> module.</p> </div> </div> <datatypes> <datatype> <name name="device" /> <div class="description"> <p>An IO device. Either <code>standard_io</code>, <code>standard_error</code>, a registered name, or a pid handling IO protocols (returned from <a href="../kernel/file.html#open/2" class="seealso">file:open/2</a>).</p> </div> </datatype> <datatype> <name name="opt_pair" /> </datatype> <datatype> <name name="expand_fun" /> </datatype> <datatype> <name name="encoding" /> </datatype> <datatype> <name name="setopt" /> </datatype> <datatype> <name name="format" /> </datatype> <datatype> <name name="location" /> </datatype> <datatype> <name name="prompt" /> </datatype> <datatype> <name name="server_no_data" /> <div class="description"><p>What the I/O-server sends when there is no data.</p></div> </datatype> </datatypes> <div class="functions"><h4>Functions</h4><hr /> <div class="function"> <h3 id="columns/0">columns/0</h3> <h3 id="columns/1">columns/1</h3> <div class="description"> <p>Retrieves the number of columns of the <code><anno>IoDevice</anno></code> (i.e. the width of a terminal). This is done by supplying <code>+pc &lt;range&gt;</code> when starting Erlang.</p> <p>Currently the only valid values for <code>&lt;range&gt;</code> are <code>latin1</code> and <code>unicode</code>. <code>latin1</code> means that only code points below 256 (with the exception of control characters etc) will be considered printable. <code>unicode</code> means that all printable characters in all unicode character ranges are considered printable by the io functions.</p> <p>By default, Erlang is started so that only the <code>latin1</code> range of characters will indicate that a list of integers is a string.</p> <p>The simplest way to utilize the setting is to call <a href="io_lib.html#printable_list/1" class="seealso">io_lib:printable_list/1</a>, which will use the return value of this function to decide if a list is a string of printable characters or not.</p> <div class="note"><h2>Note!</h2><p>In the future, this function may return more values and ranges. It is recommended to use the io_lib:printable_list/1 function to avoid compatibility problems.</p></div> </div> </div> <div class="function"> <h3 id="setopts/1">setopts/1</h3> <h3 id="setopts/2">setopts/2</h3> <div class="description"> <p>Set options for the standard IO device (<code><anno>IoDevice</anno></code>).</p> <p>Possible options and values vary depending on the actual IO device. For a list of supported options and their current values on a specific IO device, use the <a href="#getopts/1" class="seealso">getopts/1</a> function.</p> <p>The options and values supported by the current OTP IO devices are:</p> <taglist> <dt><code>binary, list or {binary, boolean()}</code></dt> <item> <p>If set in binary mode (<code>binary</code> or <code>{binary, true}</code>), the I/O server sends binary data (encoded in UTF-8) as answers to the <code>get_line</code>, <code>get_chars</code> and, if possible, <code>get_until</code> requests (see the I/O protocol description in <a href="io_protocol.html" class="seealso">STDLIB User's Guide</a> for details). The immediate effect is that <code>get_chars/2,3</code> and <code>get_line/1,2</code> return UTF-8 binaries instead of lists of chars for the affected IO device.</p> <p>By default, all IO devices in OTP are set in list mode, but the I/O functions can handle any of these modes and so should other, user written, modules behaving as clients to I/O-servers.</p> <p>This option is supported by the standard shell (<code>group.erl</code>), the 'oldshell' (<code>user.erl</code>) and the file I/O servers.</p> </item> <dt><code>{echo, boolean()}</code></dt> <item> <p>Denotes if the terminal should echo input. Only supported for the standard shell I/O-server (<code>group.erl</code>)</p> </item> <dt><code>{expand_fun, expand_fun()}</code></dt> <item> <p>Provide a function for tab-completion (expansion) like the Erlang shell. This function is called when the user presses the TAB key. The expansion is active when calling line-reading functions such as <code>get_line/1,2</code>.</p> <p>The function is called with the current line, upto the cursor, as a reversed string. It should return a three-tuple: <code>{yes|no, string(), [string(), ...]}</code>. The first element gives a beep if <code>no</code>, otherwise the expansion is silent, the second is a string that will be entered at the cursor position, and the third is a list of possible expansions. If this list is non-empty, the list will be printed and the current input line will be written once again.</p> <p>Trivial example (beep on anything except empty line, which is expanded to <code>"quit"</code>):</p> <pre class="sh_erlang"> fun("") -&gt; {yes, "quit", []}; (_) -&gt; {no, "", ["quit"]} end</pre> <p>This option is supported by the standard shell only (<code>group.erl</code>).</p> </item> <dt><code>{encoding, latin1 | unicode}</code></dt> <item> <p>Specifies how characters are input or output from or to the actual IO device, implying that i.e. a terminal is set to handle Unicode input and output or a file is set to handle UTF-8 data encoding.</p> <p>The option <em>does not</em> affect how data is returned from the I/O functions or how it is sent in the I/O-protocol, it only affects how the IO device is to handle Unicode characters towards the "physical" device.</p> <p>The standard shell will be set for either Unicode or latin1 encoding when the system is started. The actual encoding is set with the help of the <code>LANG</code> or <code>LC_CTYPE</code> environment variables on Unix-like system or by other means on other systems. The bottom line is that the user can input Unicode characters and the IO device will be in <code>{encoding, unicode}</code> mode if the IO device supports it. The mode can be changed, if the assumption of the runtime system is wrong, by setting this option.</p> <p>The IO device used when Erlang is started with the "-oldshell" or "-noshell" flags is by default set to latin1 encoding, meaning that any characters beyond codepoint 255 will be escaped and that input is expected to be plain 8-bit ISO-latin-1. If the encoding is changed to Unicode, input and output from the standard file descriptors will be in UTF-8 (regardless of operating system).</p> <p>Files can also be set in <code>{encoding, unicode}</code>, meaning that data is written and read as UTF-8. More encodings are possible for files, see below.</p> <p><code>{encoding, unicode | latin1}</code> is supported by both the standard shell (<code>group.erl</code> including <code>werl</code> on Windows®), the 'oldshell' (<code>user.erl</code>) and the file I/O servers.</p> </item> <dt><code>{encoding, utf8 | utf16 | utf32 | {utf16,big} | {utf16,little} | {utf32,big} | {utf32,little}}</code></dt> <item> <p>For disk files, the encoding can be set to various UTF variants. This will have the effect that data is expected to be read as the specified encoding from the file and the data will be written in the specified encoding to the disk file.</p> <p><code>{encoding, utf8}</code> will have the same effect as <code>{encoding, unicode}</code> on files.</p> <p>The extended encodings are only supported on disk files (opened by the <a href="../kernel/file.html#open/2" class="seealso">file:open/2</a> function)</p> </item> </taglist> </div> </div> <div class="function"> <h3 id="write/1">write/1</h3> <h3 id="write/2">write/2</h3> <div class="description"> <p>Writes the term <code><anno>Term</anno></code> to the standard output (<code><anno>IoDevice</anno></code>).</p> </div> </div> <div class="function"> <h3 id="read/1">read/1</h3> <h3 id="read/2">read/2</h3> <type name="server_no_data" /> <div class="description"> <p>Reads a term <code><anno>Term</anno></code> from the standard input (<code><anno>IoDevice</anno></code>), prompting it with <code><anno>Prompt</anno></code>. It returns:</p> <taglist> <dt><code>{ok, <anno>Term</anno>}</code></dt> <item> <p>The parsing was successful.</p> </item> <dt><code>eof</code></dt> <item> <p>End of file was encountered.</p> </item> <dt><code>{error, <anno>ErrorInfo</anno>}</code></dt> <item> <p>The parsing failed.</p> </item> <dt><code>{error, <anno>ErrorDescription</anno>}</code></dt> <item> <p>Other (rare) error condition, for instance <code>{error, estale}</code> if reading from an NFS file system.</p> </item> </taglist> </div> </div> <div class="function"> <h3 id="read/3">read/3</h3> <h3 id="read/4">read/4</h3> <type name="server_no_data" /> <div class="description"> <p>Reads a term <code><anno>Term</anno></code> from <code><anno>IoDevice</anno></code>, prompting it with <code><anno>Prompt</anno></code>. Reading starts at location <code><anno>StartLocation</anno></code>. The argument <code><anno>Options</anno></code> is passed on as the <code>Options</code> argument of the <code>erl_scan:tokens/4</code> function. It returns:</p> <taglist> <dt><code>{ok, Term, <anno>EndLocation</anno>}</code></dt> <item> <p>The parsing was successful.</p> </item> <dt><code>{eof, <anno>EndLocation</anno>}</code></dt> <item> <p>End of file was encountered.</p> </item> <dt><code>{error, <anno>ErrorInfo</anno>, <anno>ErrorLocation</anno>}</code></dt> <item> <p>The parsing failed.</p> </item> <dt><code>{error, <anno>ErrorDescription</anno>}</code></dt> <item> <p>Other (rare) error condition, for instance <code>{error, estale}</code> if reading from an NFS file system.</p> </item> </taglist> </div> </div> <div class="function"> <h3 id="fwrite/1">fwrite/1</h3> <h3 id="fwrite/2">fwrite/2</h3> <h3 id="fwrite/3">fwrite/3</h3> <h3 id="format/1">format/1</h3> <h3 id="format/2">format/2</h3> <h3 id="format/3">format/3</h3> <div class="description"> <p>Writes the items in <code><anno>Data</anno></code> (<code>[]</code>) on the standard output (<code><anno>IoDevice</anno></code>) in accordance with <code><anno>Format</anno></code>. <code><anno>Format</anno></code> contains plain characters which are copied to the output device, and control sequences for formatting, see below. If <code><anno>Format</anno></code> is an atom or a binary, it is first converted to a list with the aid of <code>atom_to_list/1</code> or <code>binary_to_list/1</code>.</p> <pre class="sh_erlang"> 1&gt; <code>io:fwrite("Hello world!~n", []).</code> Hello world! ok</pre> <p>The general format of a control sequence is <code>~F.P.PadModC</code>. The character <code>C</code> determines the type of control sequence to be used, <code>F</code> and <code>P</code> are optional numeric arguments. If <code>F</code>, <code>P</code>, or <code>Pad</code> is <code>*</code>, the next argument in <code>Data</code> is used as the numeric value of <code>F</code> or <code>P</code>.</p> <p><code>F</code> is the <code>field width</code> of the printed argument. A negative value means that the argument will be left justified within the field, otherwise it will be right justified. If no field width is specified, the required print width will be used. If the field width specified is too small, then the whole field will be filled with <code>*</code> characters.</p> <p><code>P</code> is the <code>precision</code> of the printed argument. A default value is used if no precision is specified. The interpretation of precision depends on the control sequences. Unless otherwise specified, the argument <code>within</code> is used to determine print width.</p> <p><code>Pad</code> is the padding character. This is the character used to pad the printed representation of the argument so that it conforms to the specified field width and precision. Only one padding character can be specified and, whenever applicable, it is used for both the field width and precision. The default padding character is <code>' '</code> (space).</p> <p><code>Mod</code> is the control sequence modifier. It is either a single character (currently only <code>t</code>, for Unicode translation, and <code>l</code>, for stopping <code>p</code> and <code>P</code> from detecting printable characters, are supported) that changes the interpretation of Data.</p> <p>The following control sequences are available:</p> <taglist> <dt><code>~</code></dt> <item> <p>The character <code>~</code> is written.</p> </item> <dt><code>c</code></dt> <item> <p>The argument is a number that will be interpreted as an ASCII code. The precision is the number of times the character is printed and it defaults to the field width, which in turn defaults to 1. The following example illustrates:</p> <pre class="sh_erlang"> 1&gt; <code>io:fwrite("|~10.5c|~-10.5c|~5c|~n", [$a, $b, $c]).</code> | aaaaa|bbbbb |ccccc| ok</pre> <p>If the Unicode translation modifier (<code>t</code>) is in effect, the integer argument can be any number representing a valid Unicode codepoint, otherwise it should be an integer less than or equal to 255, otherwise it is masked with 16#FF:</p> <pre class="sh_erlang"> 2&gt; <code>io:fwrite("~tc~n",[1024]).</code> \x{400} ok 3&gt; <code>io:fwrite("~c~n",[1024]).</code> ^@ ok</pre> </item> <dt><code>f</code></dt> <item> <p>The argument is a float which is written as <code>[-]ddd.ddd</code>, where the precision is the number of digits after the decimal point. The default precision is 6 and it cannot be less than 1.</p> </item> <dt><code>e</code></dt> <item> <p>The argument is a float which is written as <code>[-]d.ddde+-ddd</code>, where the precision is the number of digits written. The default precision is 6 and it cannot be less than 2.</p> </item> <dt><code>g</code></dt> <item> <p>The argument is a float which is written as <code>f</code>, if it is &gt;= 0.1 and &lt; 10000.0. Otherwise, it is written in the <code>e</code> format. The precision is the number of significant digits. It defaults to 6 and should not be less than 2. If the absolute value of the float does not allow it to be written in the <code>f</code> format with the desired number of significant digits, it is also written in the <code>e</code> format.</p> </item> <dt><code>s</code></dt> <item> <p>Prints the argument with the string syntax. The argument is, if no Unicode translation modifier is present, an <code>iolist()</code>, a <code>binary()</code>, or an <code>atom()</code>. If the Unicode translation modifier (<code>t</code>) is in effect, the argument is <code>unicode:chardata()</code>, meaning that binaries are in UTF-8. The characters are printed without quotes. The string is first truncated by the given precision and then padded and justified to the given field width. The default precision is the field width.</p> <p>This format can be used for printing any object and truncating the output so it fits a specified field:</p> <pre class="sh_erlang"> 1&gt; <code>io:fwrite("|~10w|~n", [{hey, hey, hey}]).</code> |**********| ok 2&gt; <code>io:fwrite("|~10s|~n", [io_lib:write({hey, hey, hey})]).</code> |{hey,hey,h| 3&gt; <code>io:fwrite("|~-10.8s|~n", [io_lib:write({hey, hey, hey})]).</code> |{hey,hey | ok</pre> <p>A list with integers larger than 255 is considered an error if the Unicode translation modifier is not given:</p> <pre class="sh_erlang"> 4&gt; <code>io:fwrite("~ts~n",[[1024]]).</code> \x{400} ok 5&gt; <code>io:fwrite("~s~n",[[1024]]).</code> ** exception exit: {badarg,[{io,format,[&lt;0.26.0&gt;,"~s~n",[[1024]]]}, ...</pre> </item> <dt><code>w</code></dt> <item> <p>Writes data with the standard syntax. This is used to output Erlang terms. Atoms are printed within quotes if they contain embedded non-printable characters, and floats are printed accurately as the shortest, correctly rounded string.</p> </item> <dt><code>p</code></dt> <item> <p>Writes the data with standard syntax in the same way as <code>~w</code>, but breaks terms whose printed representation is longer than one line into many lines and indents each line sensibly. It also tries to detect lists of printable characters and to output these as strings. The Unicode translation modifier is used for determining what characters are printable. For example:</p> <pre class="sh_erlang"> 1&gt; <code>T = [{attributes,[[{id,age,1.50000},{mode,explicit},</code> <code>{typename,"INTEGER"}], [{id,cho},{mode,explicit},{typename,'Cho'}]]},</code> <code>{typename,'Person'},{tag,{'PRIVATE',3}},{mode,implicit}].</code> ... 2&gt; <code>io:fwrite("~w~n", [T]).</code> [{attributes,[[{id,age,1.5},{mode,explicit},{typename, [73,78,84,69,71,69,82]}],[{id,cho},{mode,explicit},{typena me,'Cho'}]]},{typename,'Person'},{tag,{'PRIVATE',3}},{mode ,implicit}] ok 3&gt; <code>io:fwrite("~62p~n", [T]).</code> [{attributes,[[{id,age,1.5}, {mode,explicit}, {typename,"INTEGER"}], [{id,cho},{mode,explicit},{typename,'Cho'}]]}, {typename,'Person'}, {tag,{'PRIVATE',3}}, {mode,implicit}] ok</pre> <p>The field width specifies the maximum line length. It defaults to 80. The precision specifies the initial indentation of the term. It defaults to the number of characters printed on this line in the <code>same</code> call to <code>io:fwrite</code> or <code>io:format</code>. For example, using <code>T</code> above:</p> <pre class="sh_erlang"> 4&gt; <code>io:fwrite("Here T = ~62p~n", [T]).</code> Here T = [{attributes,[[{id,age,1.5}, {mode,explicit}, {typename,"INTEGER"}], [{id,cho}, {mode,explicit}, {typename,'Cho'}]]}, {typename,'Person'}, {tag,{'PRIVATE',3}}, {mode,implicit}] ok</pre> <p>When the modifier <code>l</code> is given no detection of printable character lists will take place. For example:</p> <pre class="sh_erlang"> 5&gt; <code>S = [{a,"a"}, {b, "b"}].</code> 6&gt; <code>io:fwrite("~15p~n", [S]).</code> [{a,"a"}, {b,"b"}] ok 7&gt; <code>io:fwrite("~15lp~n", [S]).</code> [{a,[97]}, {b,[98]}] ok</pre> <p>Binaries that look like UTF-8 encoded strings will be output with the string syntax if the Unicode translation modifier is given:</p> <pre class="sh_erlang"> 9&gt; <code>io:fwrite("~p~n",[[1024]]).</code> [1024] 10&gt; <code>io:fwrite("~tp~n",[[1024]]).</code> "\x{400}" 11&gt; <code>io:fwrite("~tp~n", [&lt;&lt;128,128&gt;&gt;]).</code> &lt;&lt;128,128&gt;&gt; 12&gt; <code>io:fwrite("~tp~n", [&lt;&lt;208,128&gt;&gt;]).</code> &lt;&lt;"\x{400}"/utf8&gt;&gt; ok</pre> </item> <dt><code>W</code></dt> <item> <p>Writes data in the same way as <code>~w</code>, but takes an extra argument which is the maximum depth to which terms are printed. Anything below this depth is replaced with <code>...</code>. For example, using <code>T</code> above:</p> <pre class="sh_erlang"> 8&gt; <code>io:fwrite("~W~n", [T,9]).</code> [{attributes,[[{id,age,1.5},{mode,explicit},{typename,...}], [{id,cho},{mode,...},{...}]]},{typename,'Person'}, {tag,{'PRIVATE',3}},{mode,implicit}] ok</pre> <p>If the maximum depth has been reached, then it is impossible to read in the resultant output. Also, the <code>,...</code> form in a tuple denotes that there are more elements in the tuple but these are below the print depth.</p> </item> <dt><code>P</code></dt> <item> <p>Writes data in the same way as <code>~p</code>, but takes an extra argument which is the maximum depth to which terms are printed. Anything below this depth is replaced with <code>...</code>. For example:</p> <pre class="sh_erlang"> 9&gt; <code>io:fwrite("~62P~n", [T,9]).</code> [{attributes,[[{id,age,1.5},{mode,explicit},{typename,...}], [{id,cho},{mode,...},{...}]]}, {typename,'Person'}, {tag,{'PRIVATE',3}}, {mode,implicit}] ok</pre> </item> <dt><code>B</code></dt> <item> <p>Writes an integer in base 2..36, the default base is 10. A leading dash is printed for negative integers.</p> <p>The precision field selects base. The character <code>*</code> is an optional return suppression character. The field width parameter is used to specify base. Leading white-space characters are not skipped.</p> </item> <dt><code>-</code></dt> <item> <p>An optional sign character is expected. It must follow the Erlang floating point number syntax.</p> </item> <dt><code>s</code></dt> <item> <p>A string of non-white-space characters is read. If a field width has been specified, this number of characters are read and all trailing white-space characters are stripped. An Erlang string (list of characters) is returned.</p> <p>If Unicode translation is in effect (<code>~ts</code>), characters larger than 255 are accepted, otherwise not. This device is used when no <code>IoDevice</code> argument is specified in the above function calls. However, it is sometimes desirable to use an explicit <code>IoDevice</code> argument which refers to the default IO device. This is the case with functions that can access either a file or the default IO device. The atom <code>standard_io</code> has this special meaning. The following example illustrates this:</p> <pre class="sh_erlang"> 27&gt; <code>io:read('enter&gt;').</code> enter&gt;<code>foo.</code> {ok,foo} 28&gt; <code>io:read(standard_io, 'enter&gt;').</code> enter&gt;<code>bar.</code> {ok,bar}</pre> <p>There is always a process registered under the name of <code>user</code>. This can be used for sending output to the user.</p> </div> <div class="section"> <h4>Standard Error</h4> <p>In certain situations, especially when the standard output is redirected, access to an I/O-server specific for error messages might be convenient. Example on a Unix-like operating system:</p> <pre class="sh_erlang"> $ <code>erl -noshell -noinput -eval 'io:format(standard_error,"Error: ~s~n",["error 11"]),'\</code> <code>'init:stop().' &gt; /dev/null</code> Error: error 11</pre> </div> <div class="section"> <h4>Error Information</h4> <p>The <code>ErrorInfo</code> mentioned above is the standard <code>ErrorInfo</code> structure which is returned from all IO modules.
